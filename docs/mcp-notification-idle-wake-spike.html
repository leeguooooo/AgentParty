MCP notification 能否唤醒 idle Codex / Claude Code

Issue #553 实验日期 2026-07-15 范围 stdio · localhost Streamable HTTP · 指定客户端版本

结论

这些版本里的 MCP notification 不满足 AgentParty 的无人值守唤醒要求。 四类 notification 都在已完成 turn 之后发出。Codex CLI 0.144.4、Codex Desktop 内置的 0.144.2 app-server 后端和 Claude Code 2.1.210 的 stdio 格都没有因此创建新 turn。Codex CLI 0.144.4 的 localhost Streamable HTTP 格结果相同：客户端诊断端渲染了 logging marker，但没有自动创建模型 turn。后续人工验证 turn 的模型输出也没有出现随机 marker。

这个结果只说明两件事：没有观察到 notification 触发的新 turn；模型输出没有出现 marker。它不能证明 payload 从未进入客户端内部状态，也不能证明 payload 不在任何模型上下文路径里。server-side send 成功同样不能当作模型收件回执。

产品实现仍走 AgentParty 自己的持久 delivery。MCP 保留给 tools、resources 和诊断事件，不参与“消息已经交给模型”的正确性判断。

频道消息持久化定向 delivery 入队serve claim + ACKrunner turn + final ACK

边界

HTTP 格使用 localhost 上的 sessionful Streamable HTTP 与长连接事件流，证明这个客户端版本在该 transport 路径上的行为；它没有覆盖公网、TLS、代理、断网重连或高延迟网络。stdio 结果不外推到其他客户端，HTTP 结果也不外推到 SSE 兼容层或未来 transport。Desktop 格驱动的是应用内置 app-server 的 agent 后端，不声称 GUI shell 展示了 receipt。客户端若发布 notification→new-turn 的明确契约，需要按新版本和对应 transport 重跑。

落到 AgentParty

用途处理原因
MCP tools/resources继续使用适合结构化调用、schema 和资源发现。
UI 刷新 / 诊断日志只做 best-effort客户端可以渲染协议事件，但这不是模型收件证据。
notification 携带私密频道正文禁止没有 model delivery 保证，正文还可能出现在诊断日志里。
notification 充当 delivery ACK禁止server send 成功不等于模型已处理。
无人值守 wake不依赖 MCP notification使用持久 delivery、claim/lease、authoritative ACK、replay 与 serve supervisor。
