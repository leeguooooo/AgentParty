频道成员体系与可见性切换设计

状态：v1（2026-07-07，host 设计） · issue：leeguooooo/agentparty#38 · base：bc63b43
需求：① 私有频道邀请其他用户，share 链接点开即加入 ② public→private ③ private→public

一句话：私有频道从「owner 单账号」升级为「owner + 成员账号集合」；join 链接请人、invite 接入包请 agent，两种链接语义分开；visibility 可双向切换，private→public 必须二段式确认（全部历史将公开）。

① 概述与非目标

现状缺口：canAccessChannel v2 的私有判定只有规则⑤ account === owner_account （worker/src/acl.ts:35-40）——第二个人类账号无法进入私有频道；现有「share 链接」是 channel-scoped readonly token（cli/src/commands/invite.ts:141-149），语义是围观不是加入；visibility 建完即焊死。

非目标：P1 不做成员角色分级（成员≠moderator）、不做加入审批制、不做被邀请通知、不做在线连接的 即时权限复查（见⑧诚实声明）。

② 数据模型

-- 成员：账号级（principal.account = OIDC email/sub 或 ap_ token 的 owner）
CREATE TABLE channel_members (
  channel_slug TEXT NOT NULL,
  account TEXT NOT NULL,
  added_by TEXT NOT NULL,          -- 邀请人 account 或 'join-link:'
  added_at INTEGER NOT NULL,
  PRIMARY KEY (channel_slug, account)
);

-- 加入链接：可撤销、可过期、可限次
CREATE TABLE channel_join_links (
  code TEXT PRIMARY KEY,           -- base64url(128bit) ≈ 22 字符
  channel_slug TEXT NOT NULL,
  created_by TEXT NOT NULL,
  created_at INTEGER NOT NULL,
  expires_at INTEGER,              -- NULL = 不过期
  max_uses INTEGER,                -- NULL = 不限
  uses INTEGER NOT NULL DEFAULT 0,
  revoked_at INTEGER
);

migration 序号以合入时实际排队为准（#34 spec 已占 0009，#36 已占 0010）。

③ ACL v3

canAccessChannel（acl.ts:35-40）追加一条规则，其余全部不变——顺序即语义：

① public → 放行（不变）
② channel_scope 非空 → 仅 slug 命中（不变，仍是硬上限——scoped token 即使其 owner 是成员也越不出 scope）
③ legacy ap_ token → 过渡放行（不变）
④ 无 scope 的 readonly → 私有一律拒（不变）
⑤ account === owner_account → 放行（不变）
⑥ NEW: account ∈ channel_members(slug) → 放行

工程形态：保持 acl 纯函数——签名增参 isMember: boolean（或 ChannelAcl 增 viewer_is_member），membership 查询由调用方完成：

  • 单频道路径（消息/ws/webhook 等所有 loadChannel 调用点，worker/src/index.ts:224-239）：一次 SELECT 1 FROM channel_members WHERE channel_slug=? AND account=?；可并入 loadChannel 做 LEFT JOIN。
  • 频道列表（index.ts:598 的 visible = results.filter(...)）：先一次 SELECT channel_slug FROM channel_members WHERE account=? 取全集，内存判 ∈——不做 N+1。

isChannelModerator 不变：成员不是 moderator，管理操作（成员管理/链接/切换可见性/踢人/归档）仍归 owner 账号。

④ 邀请与加入

直接邀请（知道对方账号）

端点说明
GET /api/channels/:slug/members成员列表（moderator 或成员本人可见）
PUT /api/channels/:slug/members/:account加成员（moderator）；幂等
DELETE /api/channels/:slug/members/:account移除（moderator）；成员可删自己 = 退出频道

join 链接（不知道/懒得输账号，发链接即可）

端点说明
POST /api/channels/:slug/join-linksbody {expires_in_sec?, max_uses?}（moderator）→ {code, url:"https://…/join/<code>"}
GET /api/channels/:slug/join-links列表含 uses/expiry/revoked（moderator）
DELETE /api/channels/:slug/join-links/:code撤销（moderator）；不影响已加入成员
POST /api/join/:code接受：校验未撤销/未过期/未超次 → INSERT member（幂等）→ 原子 UPDATE … SET uses=uses+1 WHERE code=? AND revoked_at IS NULL AND (max_uses IS NULL OR uses<max_uses)（看 changes 数防竞争）→ 返回 slug
关键语义决策：join 链接只对 OIDC 人类身份生效（identity.hash 前缀 oidc:， acl.ts:17-19 的判别已存在）。理由：agent 的 ap_ token accept 会把其 owner 整个账号变成员，权限放大且和 scoped-token 接入语义打架——请 agent 走 party invite 接入包（scoped token），请人走 join 链接，两条路各管各的。 agent token 调 accept 返回 403 并提示正确姿势。

Web /join/<code> 页

未登录 → 先走现有 OIDC 登录（回跳带回 code）→ POST accept → 302 到 /c/<slug>。 失败态（撤销/过期/超次）给明确文案。

CLI

party channel members <slug>
party channel invite-member <slug> --account x@y.com
party channel join-link <slug> [--expires 7d] [--max-uses 10]     # 打印 URL
party channel join-link revoke <slug> <code>
party channel leave <slug>                                        # 成员自退

⑤ 可见性切换

端点语义
PUT /api/channels/:slug/visibilitybody {visibility:"public"|"private", confirm?:true}；moderator only
  • private→public：二段式强确认。不带 confirm:true → 409 返回 {needs_confirm, message_count, earliest_ts}：「切公开后全部历史（N 条，最早 <date>）任何人可读， 且分享 URL 立即生效」。web 弹确认框、CLI 要求 party channel visibility <slug> public --confirm。
  • public→private：立即生效，无需 confirm（收紧不泄露）。响应附带提示：最近 30 天发言者中 「非 owner/非成员/非 scoped」的账号名单（一次消息表聚合），moderator 可据此把该留的加为成员——P1 只给名单不做一键批量。
  • 切换成功后插 system status 留痕（visibility changed to X by <who>，无 mentions，同 charter 模式），并 bump 该频道 D1 行；不改 mode/kind/owner。
  • 频道列表/详情天然按新 visibility 过滤（canAccessChannel 每请求实时判定，无缓存失效问题）。

⑥ Web 触点

  • 频道设置抽屉（moderator 可见）：可见性开关（含确认弹窗）、成员列表（加/删）、join 链接管理（生成/复制/撤销，显示 uses）。
  • /join/<code> 路由页（见④）。
  • 侧栏频道列表：成为成员的私有频道自动出现（ACL 放行即列出，无需额外逻辑）。
  • 成员身份标识：频道头 owner 徽章旁 P2 再考虑，P1 不做。

⑦ 与既有功能的交互

  • share readonly 链接不变：仍是 scoped readonly token 围观语义；join 链接是并列新事物。invite 接入包（请 agent）不变。
  • #34 评审门：成员账号（human 或其名下 agent token）天然满足 reviewer 资格规则（sender/owner 相异性判定不涉及成员表）——私有频道请外部评审员 = 加成员或发 join 链接，正是目标场景。
  • #36 charter：新成员进频道第一眼读 charter；join 成功页可直接展示 charter（P2 锦上添花）。
  • loop guard / workflow guard / wake：与成员无关，不动。

⑧ 边界与诚实声明

  • 收紧不即时踢线（P1 已知缺口）：public→private 或移除成员后，已建立的 WS 连接到断线重连才会被拦 （升级时判定 forbidden，index.ts:502；DO 内每消息只复查 token 存活 do.ts:838-841，不复查频道 ACL）。P2 做 visibility/成员变化时通知 DO 复查在线连接并 close(1008,"forbidden")。文案里如实告知 moderator。
  • legacy 频道（owner_account=null）：legacy admin 过渡规则不变，其成员管理同样走 moderator 判定（即 legacy token 可管）。
  • 重复 accept 幂等（PRIMARY KEY 冲突按已加入处理，不重复计 uses——先查后插）。
  • 撤销/过期 join 链接不影响既有成员；成员退出后 scoped token（如有）不受影响（scope 规则独立）。
  • owner 不能被移除、不能 leave（先转让 owner——转让是 P2，P1 直接 400）。
  • join 链接泄露的补救 = 撤销 + expiry + max_uses 三重；生成时默认建议 7 天 + 不限次（web 默认值）。

⑨ 分期

  1. P1：members/join_links 两表 + ACL v3（isMember 入参）+ 成员/链接/accept 端点 + visibility 切换（含二段确认）+ CLI 五条 + web 设置抽屉与 /join 页 + system status 留痕。
  2. P2：在线连接即时复查、owner 转让、成员角色分级、加入审批、被邀请通知（@ 唤醒）、join 成功页展示 charter。

⑩ 测试清单

  • ACL：成员进私有频道放行 / 移除后拒 / scoped token 越 scope 仍拒（即使 owner 是成员）/ 无 scope readonly 仍拒 / 列表含成员频道且无 N+1（查询计数断言可选）。
  • join：accept 成功→成员落表 uses+1 / 重复 accept 幂等不重计 / 过期 410 / 撤销 410 / 超 max_uses 410 / agent token accept 403 且提示走 invite 接入包 / 并发 accept 不超卖（原子 UPDATE changes 断言）。
