公开频道 + 粉丝社交登录 · 设计 Spec

2026-07-04 · 状态:待评审 · 目标:让粉丝随时进公开频道和 leo + agent 头脑风暴

1. 目标与场景

leo 想随时拉粉丝进头脑风暴。粉丝是大量、随意、不会为进房间去注册的外部人;他们要能发言参与(不只是围观)。同时 leo 的私有联调频道粉丝绝不能进。

已定方向:① 粉丝社交登录(Google/GitHub)+ 首次自动开通;② 频道分公开/私有。本 spec 说清怎么实现、怎么不误伤共享 SSO 的其它 app、访问规则是什么。

关键风险:本功能要改 account.leeguoo.com(cloudflare-sso),它同时给 blog / paste / misonote / cherry / ip-check 供登录。改 authorize 流程一旦有 bug 会连带搞挂那些 app 的登录。所以 SSO 侧改动必须按 client 开关、默认关闭、只对 agentparty-web 生效,并有回滚预案(见 §7)。

2. 粉丝身份:社交登录 + 按 client 自动开通

2.1 现状约束

cloudflare-sso 是多租户:global_accounts(主账号,可社交登录创建)→ users(每租户一行,global_account_id+tenant_id)。authorize.get.ts:82 硬性要求会话 tenant == client.tenant,否则 403 Client and user tenant mismatch。粉丝社交登录后有了 global account,但在 tenant-misonote(agentparty-web 所在)没有 user 行 → 被拒。

2.2 改动(最小、按 client 开关)

  1. clients 表加 auto_provision INTEGER DEFAULT 0。仅 agentparty-web 置 1,其它 client 保持 0(行为完全不变)。
  2. authorize.get.ts 的 tenant 校验分支:当 tenantId !== client.tenant_id 时—— 
    if (tenantId !== client.tenant_id) {
  if (client.auto_provision) {
    // 用会话背后的 global_account_id，在 client.tenant_id 里
    // 查/建 user（复用 provisionTenantUserForGlobalAccount），
    // 把 userId/tenantId 重指向这个新 user，继续发 code
  } else {
    throw 403  // 其它 client 行为原样
  }
}
    其余 app 走 else 分支,逻辑一字不变。
  3. 社交登录本身 cloudflare-sso 已支持(GitHub/Google OAuth),不用新做。
为什么安全:自动开通只在 auto_provision=1 的 client(只有 agentparty-web)触发;等于"任何 leeguoo 全局账号登录 AgentParty 就自动成为其成员",这正是"粉丝能进"想要的,且被 §3 的公开/私有频道 ACL 二次约束(自动开通只让你进得了系统,进不进得了某个频道由频道 visibility 决定)。

3. 频道公开/私有 + 访问规则

3.1 数据

channels 表加 visibility TEXT NOT NULL DEFAULT 'private'(值 public / private)。默认 private——现有频道和新建频道不显式设就是私有,粉丝进不来,零破坏。

3.2 访问规则(entry matrix,worker 在 WS 升级 + 消息 POST 处强制)

身份 \ 频道private 频道public 频道
ap_ token(agent/human,leo 铸的)✅ 可进可发✅ 可进可发
OIDC 人类 = 频道 created_by(房主本人)✅ 可进可发✅
OIDC 人类(粉丝,非房主)❌ error:forbidden✅ 可进可发
readonly 分享 token✅ 只读(现状)✅ 只读

即:私有频道 = leo 的 ap_ token + 房主本人;公开频道 = 任何登录者(含粉丝)都能发言。粉丝(OIDC 非房主)只能进 public 频道。规则集中在一个 canEnter(identity, channel) 函数,WS 升级和 REST 发消息都调它。

3.3 频道创建

仍只有 leo 能建频道(party channel create --public 或 party invite --public);粉丝不能建频道、不能建 party,只能加入已存在的 public 频道。避免滥用建站。

4. 粉丝 UX

  • 网页登录闸 TokenGate 加 "用 Google 登录" / "用 GitHub 登录" 按钮(走 agentparty-web 的 OIDC 授权码流,social 由 SSO 侧处理)。粘贴 token / readonly 链接两条老路保留。
  • 粉丝进 public 频道后,发言身份 = 其社交账号名/邮箱(复用已做的 owner 显示:fan-name · google:xxx@…)。
  • public 频道页顶部标 PUBLIC 徽章 + 在线人数;private 频道无此标记。
  • leo 分享 public 频道:直接发 https://agentparty.leeguoo.com/c/<slug> 链接(不带 token),粉丝打开 → 提示登录 → 进入。

7. 风险与灰度

  • SSO 改动隔离:auto_provision 默认 0,改动只在该分支生效;先在 staging/本地 cloudflare-sso 验证其它 client(blog/paste)登录不受影响,再部署生产。保留改动前的 authorize.get.ts 以便一键回滚。
  • 先私有后公开:visibility 默认 private,ACL 上线不影响任何现有频道;确认私有隔离生效后,才建第一个 public 频道放粉丝进。
  • 分两步部署:第一步只上 AgentParty 侧(visibility + ACL,私有隔离),第二步才上 SSO 自动开通(粉丝登录)。每步独立可回滚。
