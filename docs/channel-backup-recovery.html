频道备份与恢复 Runbook

适用于单频道 DO SQLite 与 D1 元数据分裂、误部署或存储恢复。备份格式版本为 backup_version: 1。

1. 先冻结写入并采集证据

party channel reconcile <slug> --json
party channel export <slug> --out <slug>-$(date +%Y%m%d-%H%M%S).json

保存当前部署 commit、D1 time-travel 时间点、对账输出和导出文件校验和。不要在证据保存前覆盖 DO。

2. 判断权威数据源

故障权威源动作
仅 D1 元数据误改导出物中的 channel / D1 表用 D1 time-travel 恢复到确认时间点，再重新对账。
DO SQLite 丢失或损坏最近一次频道导出物新建/清空目标 DO 后，按导出表的主键顺序恢复持久表；跳过 rate、webhook queue/dead letters、wake ledger 等瞬时表。
DO 与 D1 分裂D1 配置字段；DO 消息历史先恢复 D1，再通过正常频道初始化请求重推配置；消息只从导出物恢复，禁止双向盲合并。

3. D1 time-travel

使用 Cloudflare D1 time-travel 在独立恢复库验证目标时间点；确认 schema 与行数后再替换生产库。迁移不可逆时，先回退 Worker 到与目标 schema 兼容的版本。

wrangler d1 time-travel info <database>
wrangler d1 time-travel restore <database> --timestamp=<ISO-8601>

4. DO 状态重建

导出 JSON 的 durable_object.tables 是恢复输入。恢复工具必须逐表事务写入、校验行数，并保持 messages.seq、修订游标和 meta 计数器单调。恢复期间禁止客户端连接；完成后再解除写入冻结。

5. 验收

party channel reconcile <slug>
party history --channel <slug> --limit 20
party who --channel <slug>

要求 reconcile 为 ok:true、最近消息可读、成员/角色/任务数量与备份一致，并记录恢复操作者、时间点、源文件 SHA-256 和部署 commit。

边界：当前 CLI 提供导出与对账，不提供自动覆盖生产 DO 的 restore 命令。恢复写入必须在隔离环境演练并经 owner 审批；不得把 D1 time-travel 当作 DO SQLite 的备份。
